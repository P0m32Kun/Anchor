# 扫描历史对比功能 — QA 验收测试计划

## 1. 功能概述

安全人员选择两次扫描结果进行对比，高亮展示新增/消失的资产和发现（findings）。

## 2. 验收标准

### AC1: 运行选择
- [ ] 用户可从项目运行的已完成/失败列表中选择两个 PipelineRun 作为对比基线
- [ ] 选择的两个运行不能相同
- [ ] 选择后自动触发对比（或通过"开始对比"按钮触发）

### AC2: 发现差异对比
- [ ] **新增发现**：在比较运行中存在但基线上不存在的发现（按 DedupKey 判定）以绿色/高亮标记
- [ ] **已修复/消失发现**：在基线上存在但比较运行中不存在的发现以红色/删除线标记
- [ ] **状态/严重性变更的发现**：同时存在于两次运行中但 status/severity 不同的发现，展示变化细节
- [ ] 差异结果按严重性可排序/筛选

### AC3: 资产差异对比
- [ ] **新增资产**：比较运行中新增的资产（按 NormalizedValue 判定）展示差异
- [ [ ] **消失资产**：基线中存在但比较运行中不再出现的资产
- [ ] 资产类型（domain/ip/url）过滤支持

### AC4: 统计摘要
- [ ] 展示两轮运行的总体统计对比（总发现数、各严重级别数量、总资产数、工具执行统计）
- [ ] 增量变化计数：新增 N 个发现，修复 M 个，资产变化 +X/-Y

### AC5: 错误与边界
- [ ] 选择的运行状态必须是 completed 或 failed 之一
- [ ] 如果任一运行数据为空（无发现/无资产），应展示友好提示而非崩溃
- [ ] API 返回超时/错误时前端展示错误状态

## 3. 测试场景

### 3.1 正常场景
| # | 场景 | 前置条件 | 预期结果 |
|---|------|---------|---------|
| TC1 | 同一项目的两个已完成运行对比 | 项目有 ≥2 个已完成运行，各有发现和资产 | 正确展示新增/消失/变更的发现和资产，统计摘要正确 |
| TC2 | 连续的两轮扫描（增量扫描） | 运行 A（首次全量扫描），运行 B（增量，有新资产和发现） | 正确识别新增资产/发现，旧资产仍在但无新增发现 |
| TC3 | 跨多轮扫描对比 | 运行 A（第 1 轮），运行 C（第 3 轮，中间跳过 B） | 正确计算 A→C 的差异 |

### 3.2 边界场景
| # | 场景 | 预期结果 |
|---|------|---------|
| TC4 | 对比两个完全相同的运行 | 差异结果为空（无新增、无消失、无变更） |
| TC5 | 对比两个无发现的运行 | 发现差异为空，统计摘要显示 0 变化 |
| TC6 | 对比两个无资产的运行 | 资产差异为空 |
| TC7 | 对比运行 A 和自己（A vs A） | 前端应阻止相同运行选择 |
| TC8 | 运行 A（completed）vs 运行 B（running） | 应仅允许 completed/failed 的运行被选中 |

### 3.3 异常场景
| # | 场景 | 预期结果 |
|---|------|---------|
| TC9 | 选择的运行在加载中被删除 | 展示"运行不存在"错误提示 |
| TC10 | 网络请求超时 | 展示重试按钮和超时提示 |
| TC11 | 发现/资产数据量极大（5000+） | 分页/虚拟滚动正常，不卡死浏览器 |

## 4. 测试数据类型

### 4.1 发现匹配逻辑（DedupKey）
对比引擎按 `DedupKey` 匹配发现。测试验证：
- 完全相同的 DedupKey 视为同一个发现（可比较状态变化）
- 相同 DedupKey 但不同 Severity → 标记为"严重性变更"
- 相同 DedupKey 但不同 Status → 标记为"状态变更"
- 相同 DedupKey 但不同 Title → 标记为"内容变更"

### 4.2 资产匹配逻辑（NormalizedValue）
- 相同的 NormalizedValue → 同一资产
- 仅存在于基线 → "消失资产"
- 仅存在于比较运行 → "新增资产"

## 5. 非功能性需求验证
- [ ] 页面加载时间 < 2s（1000+ 发现量级）
- [ ] 差异计算在客户端完成的情况下 500 条发现以内无感知延迟
- [ ] 新增/消失/变更分类在高亮颜色上符合无障碍对比度标准

## 6. 回归测试范围
- [ ] 现有的 RunsPage 列表功能不受影响
- [ ] 现有的 FindingsPage 查询和筛选功能不受影响
- [ ] 现有的 ListPipelineRunsByProject 接口行为不变
