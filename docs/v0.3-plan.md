# v0.3 Desktop Usability & Reliability — 执行计划

> 文档角色：历史里程碑计划，仅供回溯，不作为当前实现计划。

> **阶段定位**：不堆新功能，把现有桌面工具打磨到稳定、顺手、可信赖。
>
> **技术栈**：Tauri 2.x + React 18 + TypeScript + Tailwind CSS + Zustand + Go API/Worker
>
> **文档版本**：v1.1（已评审并修订）
>
> **创建日期**：2026-04-29
>
> **修订记录**：

## 一、阶段目标

1. **主流程可稳定跑通**：创建项目 → 导入目标 → Scope 确认 → 扫描/任务 → 资产/漏洞 → 报告导出。
2. **从"功能页面集合"升级为"可连续工作的桌面应用"**。
3. **所有关键页面具备完整状态覆盖**：loading、empty、error、success、disabled、confirm、retry。
4. **建立基础自动化测试和验收清单**，减少每次改动后的回归风险。
5. **清理明显 bug、路由遗留、API 调用不一致、交互阻断点**。

### 验收标准

一个新用户打开桌面端，不看文档，也能完成一次合法目标导入、Scope 确认、扫描启动、结果查看和报告导出；遇到错误时知道发生了什么、能否重试、下一步该做什么。界面不需要花哨，但要稳定、清楚、连续、可信。

## 二、关键架构决策

### 决策 1：路由方案 — 混合嵌套路由（方案 C）

**决策**：采用全局页面 + 项目子页面的混合路由结构。

**理由**：
- 安全测试工作台的核心操作是"对某个项目做扫描"，项目是天然的上下文边界。
- 后端 API 已经是 project-scoped（`/api/projects/:id/targets` 等），前端路由应对齐。
- Dashboard（跨项目统计）和 Settings（应用配置）确实是全局的。
- 当前扁平路由（`/targets`、`/assets`）导致页面不知道在看哪个项目的数据，后续每个页面都要自己管理 project_id，反而更乱。

**目标路由结构**：

```
/                           → Dashboard（全局概览，跨项目统计）
/settings                   → Settings（应用配置）
/workers                    → Worker 管理（全局）
/projects                   → 项目列表
/projects/new               → 创建项目
/projects/:projectId        → ProjectLayout（带 project 上下文）
  /targets                  → 目标管理
  /scope                    → Scope 确认（从 Target 拆出）
  /assets                   → 资产清单
  /runs                     → 扫描任务
  /findings                 → 漏洞发现
  /reports                  → 报告导出
```

**实现方式**：
- 新增 `<ProjectLayout>` 组件，用 `useParams()` 取 `projectId`。
- fetch 项目信息后通过 React Context 传给子页面，子页面不用关心 project_id 从哪来。
- 删除当前 `/projects/:id/assets` 等 legacy 路由（它们不使用 params，与基础路由渲染相同内容）。

**影响范围**（基于 Sprint 0.3 代码审计更新）：

| 页面 | 当前 projectId 获取方式 | 数据获取模式 | 路由改造优先级 |
|------|------------------------|-------------|---------------|
| `TargetPage` | `useParams<{id}>`（仅在 Legacy 路由 `/projects/:id/targets` 有效）| `useState` + `fetch` | **P0 — 阻塞** |
| `AssetPage` | `useParams<{id}>`（仅在 Legacy 路由有效）| `useState` + `fetch` | **P0 — 阻塞** |
| `FindingsPage` | `useParams<{id}>`（仅在 Legacy 路由有效）| `useState` + `fetch` | **P0 — 阻塞** |
| `ReportsPage` | `useParams<{id}>`（仅在 Legacy 路由有效）| `useState` + `fetch` | **P0 — 阻塞** |
| `RunsPage` | `useStore(s => s.currentProject)` | `useState` + `fetch` | **P1 — 配合改造** |
| `DashboardPage` | 不依赖 projectId | 直接 `fetch`（绕过 api.ts）| **P2 — 不动** |
| `WorkersPage` | 不依赖 projectId | 直接 `fetch`（绕过 api.ts）| **P2 — 不动** |

**核心问题**：当前 4 个一级路由（`/targets`、`/assets`、`/findings`、`/reports`）在基础路由下 `useParams` 返回 `undefined`，功能不完整。只有通过 Legacy 路由 `/projects/:id/*` 才能正常获取 projectId，造成行为不一致。

**Legacy 路由处置**：
- `/projects/:id` → 重定向到 `/projects`（ProjectPage 不读 params，无意义）
- `/projects/:id/assets` → **删除**（基础路由 `/assets` 读不到 id）
- `/projects/:id/findings` → **删除**
- `/projects/:id/reports` → **删除**

**改造顺序**：
1. 创建 `ProjectLayout` + 嵌套路由 `/projects/:projectId/*`
2. 改造 TargetPage/AssetPage/FindingsPage/ReportsPage — 删除 `useParams`，改从 ProjectLayout Context 接收 projectId
3. 改造 RunsPage — 统一从 Context 接收（替代 store 读取）
4. 删除 4 条 Legacy 路由 + 添加 Legacy 重定向兼容层
5. DashboardPage/WorkersPage 后续迁移到 store（Sprint 2.4）

#### ProjectLayout 交互规范（需在 Sprint 1 实现前敲定）

路由改造不只是加一个 `<Outlet>`，以下边界场景必须在实现前明确处理方式：

| 场景 | 处理方式 |
|------|----------|
| 访问 `/projects/nonexistent-id/targets` | ProjectLayout fetch 项目 404 → 显示"项目不存在"EmptyState + 返回项目列表按钮 |
| 项目间切换时，子页面的筛选/排序/分页状态 | **重置**。切换项目意味着换上下文，保留旧筛选会造成混淆 |
| ProjectLayout 加载项目信息中，子页面显示什么？ | 显示全页 Skeleton（标题区 + 内容区），不应等子页面自己处理 |
| 当前项目被删除后 | 轮询或 SSE 检测到 404 → 清除 currentProject → 重定向到 `/projects` + Toast 提示 |
| 当前项目时间窗口过期 | ProjectLayout 顶部显示 Warning Banner（"该项目测试窗口已过期"），但页面仍可访问（只读模式） |

### 决策 2：E2E 测试方案 — 继续使用 agent-browser

**决策**：不引入 Playwright，继续使用已有的 agent-browser `.e2e.md` 体系。

**理由**：
- 项目已有完整的 agent-browser E2E 体系：`frontend/e2e/tests/` 下有 10 个测试文件 + 模板。
- agent-browser 支持 Tauri/Electron 桌面端，Playwright 对 Tauri 需要额外配置。
- 当前阶段是桌面安全工具的验收测试，不需要跑几千条回归的大型 SaaS 测试矩阵。
- agent-browser 的"截图 + 可访问性快照 + 人工判断"模式匹配验收测试需求。
- 不引入额外测试框架的维护成本。

**三层测试策略**：

| 层级 | 工具 | 覆盖目标 | 优先级 |
|------|------|----------|--------|
| Go 单元测试 | `go test` | scope engine、report generator、parser、scoring | Sprint 0 开始补 |
| API 集成测试 | `go test` + `net/http/httptest` | 每个 endpoint 的正常/异常路径 | Sprint 2 |
| 前端类型检查 | `tsc --noEmit` | 确保零类型错误 | Sprint 1 |
| E2E 验收 | agent-browser `.e2e.md` | 主流程 happy path + 异常路径 | 持续，Sprint 4 集中验收 |

**需要补齐的 package.json 脚本**：

```json
{
  "scripts": {
    "dev": "vite",
    "build": "tsc && vite build",
    "preview": "vite preview",
    "tauri": "tauri",
    "typecheck": "tsc --noEmit"
  }
}
```

> **注**：ESLint 暂不引入（当前未安装 eslint 依赖且无配置文件）。v0.3 目标是稳定性而非代码风格，tsc strict 模式（`noUnusedLocals`、`noUnusedParameters` 已开启）已提供足够的静态检查。ESLint 留到后续阶段。

#### Tauri 桌面端 smoke test（区别于浏览器 E2E）

agent-browser E2E 跑在浏览器中，但 Tauri 打包后的应用有以下差异需要专项验收：

| 差异点 | 验收方式 |
|--------|----------|
| WebView 渲染差异（macOS WKWebView / Windows WebView2） | Tauri 打包后手动截图对比浏览器截图 |
| fetch 经过 Tauri IPC | 确认 API 调用正常（超时行为、CORS 行为可能不同） |
| 报告导出（文件系统 Tauri plugin） | 手动验证导出文件路径正确、内容完整 |
| 窗口最小尺寸 | 拖拽到最小，确认布局不崩溃 |
| 窗口关闭/重启状态保持 | 关闭后重开，确认 currentProject 从 localStorage 恢复 |

Tauri smoke test 作为 Sprint 4 验收的一部分，不要求在 CI 中自动化。

### 工作流 2：UI 统一与易用性重构

**目标**：所有页面"像一个产品"，交互模式一致。

#### 2.1 设计 Token 固化

当前项目已通过 Tailwind utility class 体系隐含了设计 Token（`text-accent-green`、`bg-brand-primary/10`、`liquid-glass` 等），实际起到了 token 作用。v0.3 不需要引入 CSS custom properties 或额外的 token 系统，只需：

1. **审计现有自定义 class**（`index.css` 中的 `@layer components`），列出现有语义色列表
2. **补充 `tailwind.config.js` 的 `theme.extend.colors`**，将自定义颜色（brand-primary、accent-green、accent-yellow、accent-red、accent-teal 等）注册为官方 Tailwind 颜色，获得完整的 shade 支持
3. **统一间距和圆角**：确认全局使用 `rounded-apple` / `rounded-apple-sm`，不出现裸 `rounded-lg` 等

**不需要**：CSS custom properties（`--color-*`）、完整的 Design Token 系统、语义化命名体系。当前 Tailwind class 即 token。

> **代码审计发现**：`tsconfig.json` 已开启 `strict: true` + `noUnusedLocals` + `noUnusedParameters`，类型检查严格度足够。`npm run typecheck`（即 `tsc --noEmit`）作为主要静态检查手段。

### 工作流 3：功能可靠性与 API 调用治理

**目标**：前端不再因 API 问题卡死或显示空白。

#### 3.1 api.ts 错误处理统一

当前 `api.ts`（313 行）已有 `APIError` 类和 `fetchJSON` 基础，需要增强：

| 场景 | 当前行为 | 目标行为 |
|------|----------|----------|
| 网络断开 | 抛出不明确的 Error | 捕获 fetch 网络错误，抛出 `APIError("网络连接失败，请检查网络")` |
| 后端返回非 JSON | `res.json()` 抛异常 | 检查 Content-Type，非 JSON 时返回状态码 + 原始文本 |
| 后端结构化错误 | 已处理 `data.error.message` | 确认后端统一返回 `{ error: { message, code } }` |
| blob 导出错误 | 未处理 | blob 下载失败时检查 response Content-Type，如果是 JSON 则解析错误信息 |
| 空响应（204） | `res.json()` 抛异常 | 检查 status 204，返回 null/undefined |
| 请求超时 | 无超时处理 | 给 fetch 加 AbortController，长请求 30s 超时 |

#### 3.2 请求生命周期管理

| 问题 | 解决方案 |
|------|----------|
| 页面切换后请求继续 | 组件 unmount 时通过 AbortController 取消未完成的请求 |
| 长轮询/刷新失控 | Zustand store 中管理刷新 interval ID，页面离开时 clearInterval |
| 重复点击 | 提交按钮加 `disabled` + loading 状态，防重复提交 |
| 并发请求冲突 | 用 AbortController 取消前一次请求，只保留最新一次 |

#### 3.3 状态型页面自动刷新

Workers、Runs 页面需要定期刷新，但不能造成请求风暴：

- 统一用 `usePolling(fetchFn, intervalMs)` hook，内部处理 AbortController。
- 默认 5s 刷新，页面不可见时（`document.visibilitychange`）暂停。
- 请求失败时指数退避（5s → 10s → 20s → 40s），最多 60s。

#### 3.4 后端 API 错误格式审计

**代码审计结论**：后端错误格式已统一。

```go
// internal/errors/errors.go — 所有 handler 统一使用此结构
type AppError struct {
    Code    ErrorCode `json:"code"`
    Message string    `json:"message"`
    Detail  string    `json:"detail,omitempty"`
}

// internal/api/handlers.go — 统一输出 {"error": {"code":"...", "message":"..."}}
func writeError(w http.ResponseWriter, status int, err *errors.AppError) { ... }
```

这与前端 `api.ts` 期望的 `{ error: { message, code } }` 完全匹配。handler 中 115 处错误响应全部使用了 `writeError`。

Sprint 0 仅需确认：
- blob 导出端点（如报告下载）的错误处理方式（blob 响应不宜用 JSON 错误体）
- 是否有未使用 `writeError` 的直接 `http.Error` 调用

产出 `docs/api-error-contract.md`（简要，主要是 blob 边界的约定）。

### 工作流 4：Bug 清单与缺陷治理

**目标**：建立问题跟踪机制，确保 P0/P1 优先修复。

#### 4.1 Bug 文档结构

创建 `tasks/desktop-bugs.md`，格式如下：


### 工作流 5：测试与验收闭环

**目标**：每次改动后能快速验证没有回归。

#### 5.1 Sprint 0 立即补的脚本

```bash
# package.json scripts 补充
"typecheck": "tsc --noEmit"
```

确认 `tsconfig.json` 配置合理，`npm run typecheck` 能跑通且零错误。

#### 5.2 Go 测试补充

Go 测试应在 Sprint 0 搭建框架、Sprint 1 完成核心覆盖，不应等到 Sprint 4。这样 Sprint 2-3 的前后端改动有回归保护。

核心模块需要单元测试覆盖：

| 模块 | 路径 | 测试重点 | 时机 |
|------|------|----------|------|
| Scope Engine | `internal/scope/` | allow/deny 判定、CIDR 匹配、通配符 | Sprint 0-1 |
| Report Generator | `internal/report/` | Markdown/JSON 输出、空 findings、特殊字符 | Sprint 1 |
| Parser | `internal/parser/` | 目标导入格式解析、边界输入 | Sprint 1 |
| Scoring | `internal/scoring/` | 严重级别计算 | Sprint 1 |
| Workflow | `internal/workflow/` | 任务状态机转换 | Sprint 2 |
| API Handlers | `internal/api/` | endpoint 集成测试（httptest） | Sprint 2 |

#### 5.3 E2E 测试完善

基于已有的 `frontend/e2e/tests/*.e2e.md`，每个核心页面至少覆盖：

| 测试项 | 说明 |
|--------|------|
| 页面可打开 | navigate 后 screenshot 非空白 |
| 空状态 | 无数据时显示 EmptyState 而非崩溃 |
| API 错误状态 | 后端不可用时显示错误提示而非白屏 |
| 主要操作成功 | 创建/导入/启动等 happy path |
| 表单校验 | 必填项为空时的提示 |
| 路由跳转 | 操作成功后跳转到正确页面 |

#### 5.4 合并前检查清单

每次阶段合并前必须通过：

```bash
go test ./...
go vet ./...
npm run typecheck
npm run build
# 手动：agent-browser 跑主流程 smoke test（浏览器）
# 手动：Tauri 打包后桌面 smoke test（安装 .app/.exe，验证主流程）
```

## 八、待评审问题（含建议答案）

以下问题经代码审计后已给出建议答案，供其他评审模型交叉验证。

1. **路由方案 C（混合嵌套）** 是否合理？
   - ✅ **建议：合理**。但需在实现前确认 ProjectLayout 交互规范（5 个边界场景）

2. **不引入 Playwright，继续用 agent-browser** 是否可接受？
   - ✅ **建议：接受**。当前阶段是桌面工具验收测试，不需要大规模回归矩阵。后续需要 CI 自动化时再加 Playwright，现在引入是过度工程。

3. **Sprint 节奏和耗时估算** 是否合理？
   - ✅ **基本合理**。经代码审计后微调：Sprint 0 从 1-2 天 → 1.5-2 天（增加 SSE 验证 + Go 测试起步），Sprint 1 从 3-5 天 → 2.5-4 天（组件现状好于预期），Sprint 2 从 4-6 天 → 5-7 天（增加 workflow 测试，并发有限）。总计约 **16-24.5 天**（中位 20 天）

4. **Zustand store 改造方案**：统一管理还是页面各自管理 loading/error？
   - ✅ **建议：分层管理**。数据加载状态（如 projects 列表的 loading/error）放 store，跨页面共享。表单提交状态（如 Target 导入的提交 loading）放页面组件自身。两者不冲突。

5. **Scope 确认应该独立页面还是 TargetPage 内的 tab？**
   - ✅ **建议：独立页面（`/projects/:id/scope`）**。Scope 确认是整个扫描流程的关键决策节点，如果是 tab 容易被用户忽略。独立页面可以展示 Dry Run 结果的完整表格，提供批量 allow/deny 操作。

6. **设计 Token 要做到什么程度？**
   - ✅ **建议：只做 Tailwind theme.extend.colors 官方化**。当前项目已在用自定义 Tailwind class 充当 Token（`text-accent-green` 等），只需补 `tailwind.config.js` 注册，不需要 CSS custom properties 或完整 Design Token 系统。
