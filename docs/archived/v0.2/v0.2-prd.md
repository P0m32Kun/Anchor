# 第二阶段 PRD:实战可用化与体验提升

## 0. 阶段定位

第二阶段的目标不是继续堆更多扫描能力,而是把第一阶段已经跑通的本地闭环,升级为可以在真实授权测试项目中稳定使用的工作台。

一句话目标:

> 让工具从"能完成一次流程"变成"红队/渗透测试人员愿意在项目中持续打开、持续使用、持续沉淀证据的实战工作台"。

第二阶段围绕四个关键词设计:

1. **实战环境可用**:同时支持外网、内网和受限网络场景,远程 Worker、网络画像、代理、任务恢复、工具版本和模板版本可控。
2. **实用性提升**:资产增量扫描、Web 截图(Rod)、目录/URL 发现、工具模板、复测、误报沉淀。
3. **易用性提升**:更少配置、更清晰状态、更快验证、更少误操作。
4. **前端美观与专业感**:密度更高、层次更清晰、暗色优先、面向安全人员的操作型界面。

## 1. 红队成员视角

红队或授权测试人员在实战中最关心的不是"工具支持多少",而是:

- 我能不能快速确认这个项目当前扫描到了什么?
- 哪些资产最值得看?
- 哪些发现最可信?
- 哪些任务失败了,失败原因是什么?
- 我换网络、换节点、换代理后能不能继续跑?
- 我能不能快速把一个发现丢到 Burp/Yakit/ZAP 里手工验证?
- 项目结束时,证据、命令、截图、时间线能不能自动整理?

因此第二阶段要优先解决"项目执行现场"的问题,而不是做复杂平台化。

### 1.1 红队视角的核心场景

#### 场景 A:外网项目快速摸底

用户希望:

1. 导入客户授权域名。
2. 选择"外网标准摸底"工具模板。
3. 选择本地或远程节点。
4. 先 dry-run 看范围和预计扫描量。
5. 确认后运行。
6. 在 Dashboard 中看到存活资产、开放端口、高优先级发现、失败任务。
7. 对高优先级 Finding 做人工确认。
8. 导出阶段报告。

第二阶段必须让这个流程少配置、少跳转、状态明确。

#### 场景 B:受限网络/客户内网环境

用户希望:

1. 在客户授权环境中部署一个 Worker。
2. 桌面端连接 Worker。
3. Worker 自动上报可用工具、网络连通性、代理配置状态。
4. 扫描任务在客户网络内执行。
5. 结果回传到桌面端统一展示。

这要求第二阶段实现远程 Worker,但必须保持最小可用,不急着做多租户和复杂调度。

> v0.2 的远程 Worker 用于外网场景（云主机），内网场景通过本地 Worker 解决（笔记本带进现场，Worker 和桌面端在一起）。不支持 NAT 穿透（桌面端和 Worker 之间必须网络可达）。

#### 场景 C:企业内网安全检测

用户希望:

1. 在办公网、生产旁路区、测试区或专用扫描区部署 Worker。
2. 按网段、资产清单、业务系统分组配置授权范围。
3. 选择"内网慢扫"工具模板。
4. 识别存活主机、开放端口、Web 服务、常见中间件、弱暴露管理面和高风险漏洞线索。
5. 对高风险发现做人工验证和复测。
6. 输出内网风险清单、资产暴露面和整改复测报告。

内网检测的重点不是"打一条攻击链",而是帮助安全团队持续识别暴露面、脆弱服务、错误配置、未修复漏洞和整改状态。

#### 场景 D:项目中途复测

用户希望:

1. 对某次扫描的 confirmed Finding 发起复测。
2. 系统复用原始目标、模板、参数和范围校验。
3. 复测结果追加到原 Finding 的时间线中。
4. 报告中显示"已修复 / 仍存在 / 无法确认"。

复测是实战交付里非常高频的需求,第二阶段值得做。

#### 场景 E:手工验证接力

用户希望:

1. 在 Finding 详情页查看请求、响应、matcher、截图。
2. 一键复制 raw request。
3. 一键导出 Burp/Yakit/ZAP 可导入格式,或打开外部工具入口。
4. 手工验证后的备注、截图、结论回到 Finding。

第二阶段不需要自研 Repeater,但要把"自动发现 -> 手工验证"这一步打通。

## 2. 安全工具开发者视角

工具开发者需要关注:

- 任务执行是否可恢复;
- 错误是否结构化;
- 工具版本和模板版本是否可追溯;
- 结果解析是否抗变化;
- 日志、证据、敏感数据是否分层存储;
- UI 是否能解释系统状态,而不是把错误丢给用户;
- 远程 Worker 是否安全连接、可撤销、可审计。

### 2.1 第二阶段技术原则

1. **Worker 独立进程化**:v0.1 的同进程 Worker 升级为独立 Worker,先支持本机独立进程,再支持远程部署。
2. **网络场景抽象**:外网、内网、受限网络、复测都复用同一套数据模型,通过 NetworkProfile 和 ToolTemplate 区分执行方式。
3. **任务可恢复**:任务、步骤、artifact、checkpoint 分开记录,应用重启后能恢复状态。
4. **版本可追溯**:工具版本、模板版本、命令参数、网络画像都写入 ToolInvocation。
5. **执行环境可解释**:节点可用性、DNS、代理、出口 IP、工具缺失、模板缺失都要在 UI 中明确展示。
6. **外部工具优先**:Burp/Yakit/ZAP 继续作为外部验证工具,不自研完整 Repeater。
7. **安全边界不降级**:远程 Worker 引入后,Scope Check、授权窗口、速率限制和审计必须更严格。内网场景默认更保守,禁止默认启用高影响模板或凭据类动作。

## 3. v0.2 产品目标

### 3.1 核心目标

v0.2 验证三个能力:

1. 远程 Worker 能在真实网络环境中稳定执行扫描并回传结果。
2. 用户能更快完成资产筛选、Finding 验证、复测和报告更新。
3. 前端从"功能页集合"升级为"实战项目驾驶舱"。

### 3.2 成功标准

- 用户可以注册至少 1 个远程 Worker,并选择该 Worker 执行任务。
- Worker 页面能显示节点健康状态、工具版本、模板版本、网络画像、最近错误。
- 扫描任务中断后,系统可以明确标记可恢复/不可恢复状态。
- 支持对已确认 Finding 发起复测,并把复测证据追加到时间线。
- Findings 页面支持高效筛选、批量操作、证据预览和状态流转。
- 资产页支持按技术栈、状态码、端口、标题、风险优先级筛选。
- UI 整体进入专业工具形态:暗色优先、信息密度高、状态明确、视觉干净。

## 4. v0.2 非目标

第二阶段仍然不做:

- C2 会话操作;
- 自动利用;
- 自动横向移动;
- 自动凭据相关动作;
- 多租户 SaaS;
- 插件商店;
- 大规模分布式调度;
- 自研 Burp 类 Repeater;
- 自动绕过、规避、隐蔽通信能力。

## 5. 功能模块设计

## 5.1 实战 Dashboard

v0.1 的 Dashboard 偏项目统计,v0.2 需要变成"项目驾驶舱"。

### 展示内容

顶部状态条:

- 当前项目;
- 授权时间窗口;
- Scope 状态;
- 在线 Worker;
- 运行中任务;
- 最近错误;
- 报告完成度。

核心区域:

- 高优先级资产;
- 待验证 Finding;
- 最近新增资产;
- 失败/部分成功任务;
- 复测队列;
- 推荐下一步动作。

右侧活动流:

- 任务开始/结束;
- Finding 状态变更;
- 报告导出;
- Worker 上下线;
- Scope 修改。

### 设计原则

- 不做营销式首页。
- 不做大面积装饰图。
- 信息密度要接近专业控制台。
- 每个状态都能点击进入详情。

## 5.2 Worker 管理

### 5.2.0 内外网统一模型

平台不拆成"外网版"和"内网版"。统一模型如下:

```text
Project
  -> Target / Scope
  -> NetworkProfile
  -> WorkerNode
  -> ToolTemplate
  -> ScanTask / Finding / Evidence / Report
```

差异由三个配置决定:

- **Worker 在哪里**:公网云主机、办公网、生产旁路区、测试网、客户授权内网。
- **NetworkProfile 是什么**:external、internal、restricted、lab。
- **ToolTemplate 怎么扫**:外网快速摸底、外网标准初筛、内网慢扫、复测模式。

这样可以兼顾内外网,同时避免复制两套产品逻辑。

### 5.2.1 Worker 注册

支持两种模式:

1. 本机独立 Worker;
2. 远程 Worker。

Worker 注册信息:

- worker_id;
- name;
- endpoint;
- network_profile;
- capabilities;
- tool_versions;
- template_versions;
- max_concurrency;
- last_seen;
- status;
- trust_level。
- allowed_network_profiles。

### 5.2.2 网络画像

Worker 需要上报:

- 出口 IP;
- DNS 是否可用;
- 代理是否可用;
- 能否访问目标网络;
- 是否在内网环境;
- 当前系统平台;
- 当前工作目录是否可写。

网络画像不是为了炫技,而是为了让用户知道"为什么这个节点能扫/不能扫"。

NetworkProfile 类型:

| 类型 | 场景 | 默认策略 |
| --- | --- | --- |
| external | 公网云主机、外部扫描节点 | 外网快速摸底 / 外网标准初筛 |
| internal | 企业办公网、生产旁路区、客户内网授权区 | 内网慢扫 |
| restricted | 带代理、出口限制、访问控制严格的网络 | 内网慢扫(强制低速率 + dry-run) |
| lab | 靶场、测试环境、PoC 验证环境 | 可启用更完整模板,但仍需 Scope Check |

内网 NetworkProfile 需要额外记录:

- 网段授权来源;
- 是否允许端口扫描;
- 是否允许漏洞模板扫描;
- 是否允许截图;
- 是否允许目录/URL 发现;
- 最大并发;
- 扫描时间窗口;
- 业务系统标签;
- 生产/测试环境标记。

### 5.2.3 工具健康

每个 Worker 展示:

- Subfinder;
- httpx;
- Naabu;
- Nmap;
- Nuclei;
- Nuclei templates;
- Rod(Web 截图,需 headless Chromium);
- 可选 URL 发现工具。

状态分为:

- ready;
- missing;
- version_mismatch;
- config_error;
- permission_error。

#### 截图工具选型:Rod

Web 资产截图选用 [Rod](https://github.com/go-rod/rod)--一个基于 Chrome DevTools Protocol 的 Go 库,相比 chromedp API 更简洁,内置自动下载 Chromium 的能力。Worker 端只需 `rod.TryDownloadChromium()` 即可准备好截图环境。

选型依据:

| 方案 | 语言 | 外部依赖 | 适合? |
|------|------|---------|--------|
| Rod | Go | 自动下载 Chromium | ✅ API 最简洁 |
| chromedp | Go | 需单独安装 Chrome | ⚠️ 多一步部署 |
| playwright-go | Go | 较重,需额外 driver | ❌ 功能过度 |
| headless Chromium CLI | - | 需手写脚本管理进程 | ❌ 太原始 |

Worker 健康检查时报告 Rod 状态和 Chromium 是否可用。

### 5.2.4 Worker 安全

远程 Worker 必须满足:

- token 注册;
- token 可撤销;
- 传输加密;
- Worker 只能拉取授权任务;
- Worker 端执行前再次 Scope Check;
- Worker 不接收任意 shell 命令;
- Worker 只执行白名单工具和白名单参数模板;
- 所有命令参数脱敏后写入审计。

> 本地 Worker（内网场景）通过 localhost 通信，无需 token。远程 Worker（外网场景）必须 token 认证。

## 5.3 任务执行增强

### 5.3.1 任务步骤化

将 ScanTask 拆成 Step:

- scope_check;
- prepare_input;
- run_tool;
- collect_artifacts;
- parse_output;
- normalize_result;
- score_result;
- cleanup。

这样任务失败时,用户能看到失败在什么阶段,而不是只看到 failed。

### 5.3.2 任务恢复

支持三类恢复:

- retry:重新执行整个任务;
- resume:从最近成功 step 后继续;
- reparse:不重新扫描,只重新解析 RawArtifact。

v0.2 至少实现 retry 和 reparse,resume 可以作为 v0.3。

### 5.3.3 任务结果分级

任务状态扩展:

- completed;
- partial_success;
- failed;
- cancelled;
- scope_denied;
- needs_attention;
- stale_result。

`needs_attention` 用于提示用户:结果可用,但存在代理失败、部分目标超时、模板校验失败等问题。

## 5.4 工具模板

第二阶段要减少用户配置参数的负担。新增"工具模板"。

工具模板的核心概念非常简单--一个模板就是一份**工具清单 + 各自的速率限制**:

```
模板:「外网标准初筛」
  ├─ Subfinder  速率: 100/s
  ├─ Naabu      速率: 1000/s
  ├─ httpx      速率: 50/s
  └─ Nuclei     速率: 50/s
```

### 内置工具模板

提供 4 个预设模板覆盖主要场景,用户可在此基础上自定义(勾选/取消工具、调整速率):

1. **外网快速摸底**
   - Naabu(轻量端口)+ httpx(存活探测)+ Nuclei(仅 critical/high)
   - 默认低并发,适合快速过一遍攻击面

2. **外网标准初筛**
   - Subfinder + Naabu(常见端口)+ httpx(指纹)+ Nuclei(critical/high/medium)
   - 默认中等并发,可选开启 Web 截图

3. **内网慢扫**
   - Naabu(常见端口)+ httpx(存活、标题、技术栈)+ Nuclei(仅 critical/high,禁用高影响模板)
   - 强制低并发和时间窗口控制
   - 不做目录爆破、不扫未授权网段
   - 覆盖原来"内网低影响巡检"和"内网标准基线"两个场景,差异仅在于速率和 Nuclei 模板等级--用户自己调

4. **复测模式**
   - 只扫 confirmed / selected Finding 对应目标
   - 复用原 template、url、headers
   - 最小速率,生成复测证据追加到时间线

模板的本质是**预设值,不是约束**--用户随时可以在预设基础上调整工具开关和速率。

### 模板边界

- 模板只定义工具组合和速率,不包含自动利用。
- 用户选择模板后可以看到即将调用的工具清单和速率摘要。
- 内网场景默认更低并发、更严格 Scope Check,顶部状态条显示"内网"标签作为提醒。

## 5.5 资产体验增强

资产页从"列表"升级为"资产筛选台"。

### 需要新增

- 技术栈筛选;
- 状态码筛选;
- 端口筛选;
- 标题搜索;
- 新增/消失资产标记;
- 最近扫描时间;
- 风险优先级;
- 截图缩略图;
- 批量加入扫描;
- 批量标记业务标签。

### 红队实用视角

最值得优先展示的资产:

- 管理后台;
- 登录入口;
- 旧系统;
- 暴露中间件;
- 非标准端口 Web;
- 500/403/401 状态码资产;
- 标题或路径出现 admin、console、dashboard、swagger、actuator 等关键词的资产。

这部分只做优先级排序和提示,不做攻击动作。

### 内网资产视角

内网资产页需要额外支持:

- 网段分组;
- 业务系统分组;
- 生产/测试环境标签;
- 资产来源:导入、扫描发现、历史遗留;
- 主机存活状态;
- 常见服务聚合:RDP、SSH、SMB、数据库、中间件、Web;
- 高风险暴露面提示:未授权管理面、默认页面、过期中间件、明文服务、弱加密协议线索;
- 新增/消失资产对比;
- 同网段风险分布。

内网资产排序不应只看漏洞严重级别,还要看业务标签、网络位置、是否生产环境、是否管理面、是否暴露敏感服务。

## 5.6 Findings 验证增强

### 5.6.1 Finding 详情页

采用三栏布局:

- 左侧:Finding 列表和筛选;
- 中间:详情、证据、请求响应、截图;
- 右侧:状态流转、备注、复测、报告入口。

### 5.6.2 时间线

每个 Finding 需要时间线:

- 首次发现;
- 证据追加;
- 人工确认;
- 误报标记;
- 复测发起;
- 复测结果;
- 加入报告;
- 报告导出。

### 5.6.3 外部验证工具接力

v0.2 支持:

- 复制 raw request;
- 导出 request 文件;
- 复制 curl;
- 标记"已发送到外部工具";
- 手工上传验证截图;
- 验证备注模板。

后续再考虑与 Burp/Yakit/ZAP 做更深集成。

### 5.6.4 复测

复测对象:

- 单个 Finding;
- 批量 selected Findings;
- 某个报告中的 Findings。

复测输出:

- still_present;
- fixed;
- inconclusive;
- failed_to_test。

复测结果不覆盖原 Evidence,只追加 Evidence 和 FindingRevision。

## 5.7 报告增强

第二阶段报告重点不是更花哨,而是更像交付物。

新增能力:

- 报告模板;
- 报告章节开关;
- 风险统计图;
- 复测结论;
- 资产汇总表;
- 工具版本附录;
- 任务失败说明;

## 6. 前端体验设计

## 6.1 视觉方向

推荐方向:

- 暗色优先;
- 中性灰黑为底;
- 低饱和强调色;
- 严重性颜色仅用于风险表达;
- 表格、分栏、面板为主;
- 少用大卡片;
- 不做大面积渐变;
- 不做装饰性图形;
- 不做营销页风格。

色彩建议:

- 背景:近黑/深灰;
- 面板:比背景略浅;
- 边框:低对比灰;
- 主强调色:青蓝或绿色,表达在线、运行、可操作;
- 风险色:critical 红、high 橙红、medium 黄、low 蓝、info 灰。

### 6.2 信息架构

导航建议:

1. Dashboard
2. Targets
3. Assets
4. Runs
5. Findings
6. Reports
7. Workers
8. Settings

新增 Workers 为一级入口。

内外网切换方式:

- Project 级展示当前 NetworkProfile;
- Dashboard 顶部状态条展示"外网 / 内网 / 受限 / 靶场"标签;
- ToolTemplateSelector 根据 NetworkProfile 默认推荐模板;
- Worker 选择器只展示与当前 NetworkProfile 匹配的 Worker;
- ScopeDecisionDrawer 显示内网网段授权来源和排除规则。

### 6.3 操作模式

常用操作必须减少跳转:

- 资产列表中可直接加入扫描计划;
- Finding 列表中可直接改状态;
- 任务失败可直接重试或 reparse;
- Worker 异常可直接查看修复建议;
- Scope deny 可直接跳到匹配规则。

### 6.4 关键组件

必须建设的组件:

- StatusBadge;
- SeverityBadge;
- ConfidenceMeter;
- WorkerHealthIndicator;
- ScopeDecisionDrawer;
- RawEvidenceViewer;
- RequestResponseViewer;
- Timeline;
- CommandPreview;
- ToolTemplateSelector;
- RunStepProgress;
- FindingTriagePanel。

### 6.5 可用性标准

- 1000 条 Finding 列表滚动不卡顿。
- 表格列可配置显示/隐藏。
- 筛选条件可保存为 View。
- 所有危险操作需要二次确认。
- 空状态提供下一步动作,不只显示"暂无数据"。
- 错误状态提供修复建议,不只显示错误码。

## 9. 第二阶段里程碑

### P2-M0:体验基线与信息架构

交付:

- Workers 一级页空状态;

验收:

- 用户打开项目后 30 秒内能判断当前项目状态。
- 所有一级页有清晰空状态和下一步动作。

### P2-M1:独立 Worker

交付:

- 本机 Worker 独立进程;
- Worker 注册;
- Worker 健康检查;
- Worker 工具版本展示;
- 任务选择 Worker 执行。

验收:

- 桌面端重启后 Worker 状态可恢复。
- Worker 异常退出后 UI 能提示并允许重连。

### P2-M2:Worker 最小闭环

交付:

- 本地 Worker 独立进程（内网场景）;
- 远程 Worker token 注册 + 心跳 + 长轮询（外网场景）;
- Worker 健康检查（工具版本、Rod 状态）;
- Worker 执行前 Scope Check;
- NetworkProfile external/internal/restricted/lab 标记;
- 本地 Worker 崩溃自动重启（最多 3 次 backoff）。

验收:

- 桌面端启动后本地 Worker 自动启动并可用（内网场景）。
- 在另一台机器上启动远程 Worker 并成功注册（外网场景）。
- Worker 能完成一次外网标准初筛。
- Worker 能完成一次内网慢扫。
- token revoke 后远程 Worker 不再能接收任务。

### P2-M3:工具模板与资产筛选

交付:

- 4 个内置工具模板(外网快速摸底、外网标准初筛、内网慢扫、复测);
- 模板选择界面,展示工具清单和速率摘要;
- 用户可以自定义工具开关和速率;
- 资产筛选台;
- 新增/消失资产标记;
- 管理后台优先级提示;
- Web 资产截图缩略图(Rod)。

验收:

- 用户可以用"外网标准初筛"模板完成一次扫描,无需理解每个工具参数。
- 用户可以用"内网慢扫"模板完成一次授权网段摸底,无需手工配置每个工具参数。
- 用户可以在资产页快速筛出高价值目标。
- Web 资产自动截图,资产列表可见缩略图。

### P2-M4:验证与复测增强

验收:

## 12. 判断标准

第二阶段做完以后,应该能回答:

- 它能不能在客户授权网络中跑?
- 它能不能同时表达外网项目和内网项目?
- 节点出问题时,用户能不能知道为什么?
- 扫描中断后,证据会不会丢?
- 用户能不能快速筛出值得看的资产?
- 用户能不能快速完成 Finding 验证和复测?
- 报告和归档是否足够交付?
- 前端是否像一个专业安全工具,而不是 CRUD demo?

如果这些问题答案都是"是",第二阶段才算真正完成。

## 13. 参考依据

第二阶段设计继续坚持"成熟工具编排,不重写底层工具":

- Nuclei 官方文档强调其模板、批量扫描、输出格式、workflow 和复测能力,适合作为工具模板和复测能力的执行基础:https://docs.projectdiscovery.io/opensource/nuclei/overview
- httpx 官方仓库说明它是多探针 HTTP toolkit,支持 URL/IP/CIDR 输入、标题、状态码、CNAME、CDN、响应时间等探测;同时提示它主要是 standalone CLI,不建议直接当常驻服务使用。因此 v0.2 仍按短生命周期任务执行:https://github.com/projectdiscovery/httpx
- Naabu 官方仓库定位为快速端口扫描器,并强调与其他工具组合用于攻击面发现,适合继续作为端口发现层:https://github.com/projectdiscovery/naabu
- Rod 是基于 Chrome DevTools Protocol 的 Go 库,API 简洁且内置自动下载 Chromium,适合作为 Worker 端 Web 截图工具:https://github.com/go-rod/rod
- Tauri v2 capabilities 文档将权限控制作为桌面安全边界,v0.2 前端和本地能力调用需要继续保持最小权限:https://v2.tauri.app/security/capabilities/
