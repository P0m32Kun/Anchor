# 目标中心自动化安全测试工作台 PRD

## 0. 文档状态

- 文档类型：MVP 产品需求文档
- 当前阶段：可进入需求拆解、原型和工程排期
- 目标版本：v0.1 本地闭环 MVP
- 核心原则：目标中心、工具编排、结果归一、人工确认、证据可追溯

## 1. 一句话定位

一款面向授权安全测试的目标中心工作台，通过编排成熟开源工具、强制范围校验、统一结果模型、人工验证队列和报告生成，减少安全人员在工具切换、数据整理、证据归档和报告交付上的重复劳动。

## 2. 背景与用户痛点

安全测试人员通常不是缺少扫描工具，而是被工具之间的割裂消耗：

- 目标、资产、端口、Web 服务、漏洞、证据、报告分散在不同工具和文件中。
- Subfinder、httpx、Naabu、Nmap、Nuclei、Burp、Yakit、Word/Excel 之间频繁复制粘贴。
- 扫描结果需要人工去重、归并、判断可信度和整理证据。
- 授权范围、排除范围、时间窗口、速率限制常靠人工记忆，容易误扫。
- 报告交付需要重新组织漏洞描述、复现步骤、截图、请求响应和修复建议。
- 测试中途很难回答“哪些资产最值得看、哪些发现最可信、下一步该做什么”。

因此，本产品不重写扫描器、不重写 C2、不重写 Burp 类手工测试工具。它的价值在于把现成工具串成一个可控、可追溯、可交付的项目工作流。

## 3. 目标用户

### 3.1 MVP 优先用户

MVP 优先服务：

- 中小安全团队的渗透测试工程师；
- 安全服务商的交付人员；
- 企业内部安全团队中负责外网资产巡检和授权 Web 初筛的人员。

这些用户通常具备基础工具使用能力，但希望减少重复整理和报告时间。

### 3.2 暂不优先用户

MVP 暂不优先服务：

- 需要完整多人协作、审批流和资产生命周期管理的大型企业平台团队；
- 以高级红队后渗透为核心场景的团队；
- 需要商业漏洞管理平台完整替代品的团队；
- 非授权扫描或攻击自动化用户。

## 4. 产品目标

### 4.1 MVP 核心目标

验证一个闭环：

> 用户输入授权目标和范围后，产品自动完成资产发现、存活探测、端口识别、Web 指纹采集、Nuclei 初筛、人工确认和 Markdown/JSON 报告导出。

### 4.2 成功标准

MVP 成功标准：

- 单个项目可以从目标输入走到报告导出，无需手动整理多个工具输出。
- 所有扫描任务在执行前都经过 Scope Check。
- 每条 Finding 都能追溯到来源工具、命令参数、工具版本、原始输出和证据。
- 用户可以在验证队列中完成确认、误报、接受风险、加入报告。
- 报告中每个确认漏洞包含资产、风险、证据、复现摘要和修复建议。
- 本地模式下，100 个以内 Web 目标可以稳定完成一次标准初筛。

### 4.3 非目标

MVP 明确不做：

- 自动利用；
- 自动横向移动；
- 自动凭据转储；
- 自动投递载荷；
- C2 服务端实现；
- C2 会话操作封装；
- 绕过检测、规避防护、隐蔽通信能力产品化；
- 插件商店；
- 多租户 SaaS 管理后台；
- Neo4j 攻击图谱；
- AI Agent 自主决策攻击链；
- 替代 Burp、Yakit、DefectDojo、Dradis、AttackForge。

## 5. 产品边界

### 5.1 产品应该做

- 管理项目、目标、授权范围和排除范围。
- 根据目标和策略生成扫描计划。
- 调用现成工具执行资产发现、存活探测、端口发现、指纹识别和漏洞初筛。
- 将工具输出归一到统一数据模型。
- 保留原始输出、截图、请求响应、命令参数和工具版本。
- 基于规则给出资产优先级、Finding 可信度和下一步建议。
- 提供人工验证队列。
- 导出 Markdown 报告和机器可读 JSON。

### 5.2 产品不应该做

- 不实现自己的漏洞扫描引擎。
- 不实现自己的 C2。
- 不把高风险利用动作做成一键按钮。
- 不试图替代专业手工测试代理。
- 不把“集成工具数量”作为核心卖点。

## 9. MVP 用户流程

### 9.1 创建项目

用户操作：

1. 新建项目。
2. 填写项目名称、客户/组织、测试目的、测试时间窗口。
3. 选择测试策略：轻量、标准、深度。

系统输出：

- Project 记录；
- 默认扫描策略；
- 默认报告大纲；
- 空的审计日志。

验收标准：

- 项目未配置授权范围时，不允许启动扫描。
- 项目时间窗口过期时，不允许启动扫描。

### 9.2 配置目标与授权范围

用户操作：

1. 输入域名、URL、IP 或 CIDR。
2. 配置授权范围。
3. 配置排除范围。
4. 配置速率限制。

系统输出：

- Target 列表；
- Scope 规则；
- Exclusion 规则；
- 范围预览。

验收标准：

- 支持手动输入和文件导入。
- 支持域名、URL、IP、CIDR。
- 排除范围优先级高于包含范围。
- 未命中授权范围的目标不会进入扫描计划。

### 9.3 生成扫描计划

用户操作：

1. 选择工作流：资产发现、Web 初筛、完整 MVP 流程。
2. 查看执行计划预览。
3. 确认后启动。

系统输出：

- ScanPlan；
- ScanTask 列表；
- ScopeDecision 记录；
- 审计日志。

执行计划必须展示：

- 将扫描的目标数量；
- 排除的目标数量和原因；
- 使用工具；
- 主要参数；
- 预计强度；
- 速率限制；
- 运行节点；
- 是否在授权时间窗口内。

验收标准：

- 用户确认前不执行任何扫描命令。
- 每个任务必须有 ScopeDecision。
- 被排除目标必须展示原因。
- 支持干运行（dry-run）模式：不调用外部工具，仅输出 Scope Check 结果、计划执行的目标列表和预计耗时。

干运行模式验收标准：

- 干运行同样经过完整 Scope Check。
- 输出每一项目标的 allow/deny 及原因。
- 展示每个步骤将调用的命令（参数脱敏）。

### 9.4 执行本地 Worker

#### 进程拓扑

MVP 中 Worker 作为 Control Plane 内的 goroutine 运行（与 Control Plane 同进程），通过函数调用传递任务。v0.2 再拆分为独立进程通过 HTTP 通信。

每次任务执行使用独立的 workdir：`<data_dir>/workdirs/<project_id>/<task_id>/`，任务结束后保留 RawArtifact 但清理临时文件。

#### 执行流程

1. Worker 从 Control Plane 接收 ScanTask。
2. **执行前 Scope Check**（TOCTOU 防护）：校验 ScopeDecision 的时间戳晚于 Scope 最后修改时间；若 Scope 已变更，重新执行 Scope Check。Scope Check 失败则任务标记为 `scope_denied` 并中止。
3. 检查工具是否存在。
4. 执行工具（子进程，workdir 隔离）。
5. 收集 stdout、stderr、exit code、JSONL、截图或其他 artifact。
6. 每个工具输出大小上限 100 MB，超出部分截断并记录 TruncationWarning。
7. 将结果回传给本地 Control Plane。

#### 验收标准

- 工具缺失时给出可读错误和安装提示。
- 任务失败时保存失败原因（包括 exit code、stderr 摘要）。
- 用户可以取消正在运行的任务。
- 任务取消后，子进程必须被 SIGTERM→SIGKILL 终止。
- 不在日志中写入 API Key、Token、Cookie 等敏感信息。
- 每次任务执行前重新确认 ScopeDecision 有效（防止 TOCTOU）。

### 9.5 结果归一

系统行为：

1. 解析各工具输出。
2. 创建或更新 Asset、Port、Service、WebEndpoint。
3. 创建 Finding。
4. 保存 RawArtifact。
5. 计算 confidence 和 priority。

验收标准：

- 重复资产可以归并。
- 同一 URL 的重复 Finding 可以去重。
- 每条 Finding 能打开原始证据。
- 解析失败不导致整个任务失败，应记录 ParseError。
- Finding 去重规则（详见 10.6.1 Finding 去重规则）。

### 9.6 人工验证

用户操作：

1. 查看待验证 Finding。
2. 查看请求、响应、匹配片段、截图和命令参数。
3. 标记为 confirmed、false_positive、accepted_risk、ignored。
4. 添加备注。
5. 加入报告。

验收标准：

- 状态变更必须写入 FindingRevision。
- confirmed 的 Finding 可以进入报告。
- false_positive 默认不进入报告。
- 用户备注和截图可以成为 Evidence。

### 9.7 报告导出

用户操作：

1. 选择报告范围。
2. 调整漏洞顺序。
3. 预览 Markdown。
4. 导出 Markdown 和 JSON。

系统输出：

- Markdown 报告；
- JSON 数据包；
- 报告导出审计记录。

验收标准：

- 报告包含项目摘要、测试范围、方法说明、漏洞列表、证据、修复建议、附录。
- 每个漏洞包含资产、严重性、可信度、状态、证据和复现摘要。
- JSON 可被后续同步到 DefectDojo 等平台。

### 10.4 Worker 与错误恢复

#### 进程模型

MVP 中 Worker 为 Control Plane 内的 goroutine（同进程，函数调用）。v0.2 再拆分为独立进程。

#### Worker 能力

- 工具健康检查（详见下方）；
- 工具版本采集；
- 子进程执行（os/exec，独立 workdir）；
- 超时控制（per-task 可配置超时，默认值见策略表）；
- 取消控制（SIGTERM，宽限期后 SIGKILL）；
- 输出大小硬上限（100 MB，超出截断并记录 TruncationWarning）；
- 结果回传。

每个任务使用独立 workdir：`<data_dir>/workdirs/<project_id>/<task_id>/`。任务完成后保留 RawArtifact，清理临时文件。

#### 工具健康检查

内容包括：

- binary path 存在且可执行；
- version（通过 `--version` 或等价标志采集）；
- template path（Nuclei 专属：验证模板目录存在且 `nuclei -validate` 通过）；
- writable workdir（父目录可写）；
- network availability（出站 TCP 连接可用性，至少能解析并连接 1.1.1.1:53）；
- DNS 解析可用性（至少能解析 `example.com`）；
- proxy 可达性（若用户配置代理，验证代理可连接）；
- last_check_at。

健康检查在以下时机执行：
- 应用启动时；
- 用户手动触发；
- 每次 ScanPlan 确认前。

#### 10.4.1 错误恢复策略

| 场景 | 处理 |
| --- | --- |
| 工具返回零结果（如 Subfinder 找到 0 个域名） | 任务标记为 `completed`（非失败），后续依赖任务收到空输入后自身处理 |
| 工具对部分目标超时 | Nuclei 任务标记为 `partial_success`，超时目标单独记录 |
| 子进程被 SIGKILL | 任务标记为 `cancelled`，已收集的部分输出保留在 RawArtifact |
| 解析器无法解析某条工具输出 | 记录 ParseError（含原始行），但继续解析剩余输出 |
| 工具输出超过 100 MB | 截断并记录 TruncationWarning，已解析部分正常入库 |
| 用户手动重试失败任务 | 创建新 ScanTask，生成新 workdir，旧 RawArtifact 保留不覆盖 |
| 工具未安装 | 任务标记为 `failed`，错误信息包含安装提示 |

- redaction_status
- created_at

FindingRevision 字段：

- id
- finding_id
- from_status
- to_status
- comment
- operator

#### 10.6.1 Finding 去重规则

去重键（dedup_key）计算规则：

```
dedup_key = SHA256(template_id + "|" + url + "|" + matcher_name_or_index)
```

- 同一 URL 上不同 Nuclei template 命中不同漏洞类型（如 XSS vs SQLi）→ 不同 Finding，不去重。
- 同一 URL 上不同 template 命中同一漏洞类型 → 保留两条 Finding，在 UI 中标记为"可能重复"供人工判断。
- 同一 URL 上同一 template 两次扫描均命中 → 更新已有 Finding 的 Evidence，不创建新 Finding。
- 去重时合并 Evidence：新 Evidence 追加到已有 Finding 的 Evidence 列表，不覆盖旧 Evidence。
- 校验入口：ScanPlan 确认阶段的 Scope Check 结果为 deny 的目标不会进入扫描，因此不存在不同 Scope 下的去重问题。

### 10.7 可信度与优先级

MVP 使用规则评分，不使用 AI 自主判断。

confidence 加分因素：

- Nuclei 有明确 matcher 证据；
- 存在请求响应；
- 命中具体版本；
- 多工具结果互相印证；
- 历史误报率低。

priority 加分因素：

- severity 高；
- 资产暴露在公网；
- WebEndpoint 可访问；
- 登录入口、管理后台、敏感路径；
- 业务关键标签。

验收标准：

- 评分规则可解释。
- UI 展示主要加分原因。
- 用户可以覆盖 Finding 状态，但不直接篡改原始证据。

## 12. 扫描策略

### 12.1 轻量

- 目标：快速确认存活和高危明显问题。
- 端口：常见 Web 端口。
- Nuclei：critical/high。
- 并发：低。
- 适用：初次摸底、敏感环境、低影响巡检。

### 12.2 标准

- 目标：MVP 默认策略。
- 端口：Top 常见端口或用户配置端口组。
- Nuclei：critical/high/medium，基于指纹筛选。
- 并发：中。
- 适用：常规授权 Web 初筛。

### 12.3 深度

- 目标：更完整发现。
- 端口：用户授权范围内自定义或全端口。
- Nuclei：更多模板、更长超时。
- 并发：用户显式确认。
- 适用：明确授权的深入测试。

MVP 可以实现轻量和标准，深度只保留配置结构。

## 13. 安全与合规要求

### 13.1 强制 Scope Check

#### 校验时机

Scope Check 在两个时机执行：

1. **计划阶段**（ScanPlan 确认时）：对所有候选目标执行 Scope Check，生成 ScopeDecision。
2. **执行阶段**（Worker 执行每个任务前）：重新校验 ScopeDecision 时间戳是否晚于 Scope 最后修改时间。若不一致则重新执行 Scope Check。（TOCTOU 防护）

#### 校验规则

域名匹配：

- 精确匹配：`example.com` 匹配 `example.com`。
- 子域名包含：`example.com` 同时匹配 `sub.example.com`、`a.b.example.com`。
- 同级/父级不包含：`example.com` 不匹配 `notexample.com`。
- 通配规则：`*.example.com` 匹配 `a.example.com`、`b.example.com`，但不匹配 `a.b.example.com`。

URL 匹配：

- 前缀匹配：`https://example.com/app` 匹配 `https://example.com/app/dashboard`。
- 不包含端口差异：`https://example.com` 不匹配 `http://example.com:8080`，除非显式授权。

IP/CIDR 匹配：

- 精确 IP：`192.168.1.1` 匹配自身。
- CIDR 包含：`10.0.0.0/8` 匹配 `10.1.2.3`。

CDN / 云 WAF 场景：

- httpx 探测到的 IP 若为 CDN 节点（如 Cloudflare），Scope Check 使用原始域名进行匹配，不使用 CDN IP。若用户同时授权了 CDN IP 所在的 CIDR，不做额外处理。

排除优先：

- 排除规则优先级高于包含规则。若目标同时命中 include 和 exclude，最终判定为 deny。

速率限制：

- 速率限制分为**工具级**（如 Naabu 发包速率）和**全局级**（所有工具总发包速率）。
- MVP 先实现工具级速率限制。全局速率限制可在 Plan 层级配置预留，v0.2 再实现。
- 速率限制在 Scope Check 阶段仅做可配置性校验（值 ≥ 0），实际控速由 Worker 在工具执行时通过参数传递。

#### ScopeDecision 持久化

校验结果必须持久化为 ScopeDecision。

ScopeDecision 字段：

- id
- project_id
- target_value
- task_id
- decision: allow | deny
- matched_rule_id
- reason
- created_at

TOCTOU 标记：ScopeDecision 带时间戳。Worker 执行前检查 `ScopeDecision.created_at >= MAX(ScopeRule.updated_at)`。若条件不满足，重新执行 Scope Check 并写入新 ScopeDecision。

### 13.2 操作分级

| 等级 | 类型 | 示例 | MVP 处理 |
| --- | --- | --- | --- |
| L1 | 只读分析 | 查看结果、生成报告 | 允许 |
| L2 | 轻量探测 | httpx 存活、标题获取 | Scope Check 后允许 |
| L3 | 扫描任务 | Naabu、Nmap、Nuclei | 需要执行计划确认 |
| L4 | 验证动作 | 高危 PoC 复验 | 只进入人工队列，不做自动利用 |
| L5 | 后渗透 / C2 | 会话操作、凭据相关动作 | MVP 不实现 |

### 13.3 密钥管理

MVP 涉及的敏感信息：

- 空间测绘 API Key，若启用；
- 代理配置；
- 私有模板库凭据，若启用；
- Worker token，后续远程 Worker 使用。

要求：

- **本地加密存储**：使用操作系统原生凭据存储——macOS Keychain、Windows DPAPI（Credential Manager）、Linux freedesktop Secret Service（通过 `libsecret`）。不以明文写入 SQLite 或配置文件。
- **加密密钥派生**：若目标平台无原生凭据存储（headless Linux），回退到文件加密，密钥由用户设置的主密码通过 Argon2id 派生。
- 日志脱敏；
- 导出报告默认不包含密钥；
- RawArtifact 需要标记是否完成脱敏检查；
- 错误日志不得输出完整 Cookie、Authorization、API Key。

## 18. 风险与应对

### 18.1 工具输出变化

风险：CLI 工具版本升级导致 JSON 字段变化。

应对：

- 保存工具版本；
- 解析器按版本兼容；
- 原始输出永远保留；
- 解析失败记录 ParseError。

### 18.2 扫描误伤

风险：用户误配范围或策略过强。

应对：

- 强制 Scope Check；
- 执行计划确认；
- 深度策略二次确认；
- 速率限制；
- 时间窗口控制。

### 18.3 敏感信息泄露

风险：请求响应、日志、报告中含 Token、Cookie、API Key。

应对：

- 日志脱敏；
- 报告导出前提示敏感字段检查；
- RawArtifact 标记 redaction_status；
- 默认隐藏 Authorization、Cookie、Set-Cookie。

### 18.4 产品范围膨胀

风险：过早加入 C2、图谱、多节点、AI 自动决策。

应对：

- v0.1 只做本地闭环；
- 所有新功能必须证明能减少用户工作量；
- 远程 Worker 进入 v0.2；
- C2 只读适配最早进入 v0.5。
